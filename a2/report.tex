\section{Introduction}

Parallel computing enables large-scale computations to complete in less time by distributing work across multiple execution units. This assignment explores two algorithms that are amenable to parallelization:

\begin{enumerate}
    \item \textbf{Odd-Even Transposition Sort}: A comparison-based sorting algorithm with a
          regular, phase-alternating structure where independent pairs can be processed in parallel.
    \item \textbf{Pearson Correlation Coefficient (PCC)}: A statistical measure of linear
          correlation between pairs of data rows. Computing all $\frac{m(m-1)}{2}$ pair-wise
          correlations for an $m \times n$ matrix is computationally intensive and highly
          parallelisable.
\end{enumerate}

For each algorithm a correct sequential reference implementation is provided
(\texttt{oddevensort.cpp} and \texttt{pcc\_seq.cpp}). Parallel versions were developed on top of
these references without modifying them.

\section{Odd-Even Transposition Sort}

\subsection{Single-block}

\subsubsection{Parallelization Strategy}

The single-block kernel \texttt{oddeven\_sort\_singleblock} assigns the entire sort to a single CUDA block. All $n$ phases are executed inside of one kernel invocation. Within each phase, every thread handles a strided set of compare and swap pairs:

\begin{lstlisting}[language=C++, caption={Single-block kernel}]
__global__ void oddeven_sort_singleblock(int *d_data, int n) {
    int tid    = threadIdx.x;
    int stride = blockDim.x;

    for (int phase = 0; phase < n; phase++) {
        int start = phase & 1;
        for (int j = start + 2 * tid; j < n - 1; j += 2 * stride) {
            if (d_data[j] > d_data[j + 1]) {
                int tmp       = d_data[j];
                d_data[j]     = d_data[j + 1];
                d_data[j + 1] = tmp;
            }
        }
        __syncthreads();
    }
}
\end{lstlisting}

Thread \texttt{tid} handles pairs at indices $\text{start} + 2\cdot\text{tid},\;
\text{start} + 2\cdot(\text{tid}+\text{stride})$ and so on. This distributes the $\lfloor n/2 \rfloor$ active pairs across all threads in a strided fashion.

\subsubsection{Synchronization}

Synchronization between phases is achieved with \texttt{\_\_syncthreads()}, which acts as a barrier. All threads in the block must complete their compare and swap work for phase $k$ before any thread may begin phase $k+1$. Because \texttt{\_\_syncthreads()} is only valid within a single block, this strategy is limited to a maximum of 1024 threads.

\subsubsection{Data Decomposition}

Each thread covers pairs in a strided pattern across the active index range. This distributes work evenly when $n/2$ is a multiple of the thread count. The stride layout sacrifices some spatial locality but simplifies the index arithmetic while still handling the remainder edge case.

\subsection{Multi-block}

\subsubsection{Parallelization Strategy}

The multi-block implementation \texttt{oddeven\_phase\_multiblock()} launches a separate kernel for each of the $n$ phases. Each kernel assigns exactly one compare and swap pair to each thread.

\begin{lstlisting}[caption={Multi-block kernel -- one pair per thread}]
__global__ void oddeven_phase_multiblock(int *d_data, int n, int phase) {
    int idx   = blockIdx.x * blockDim.x + threadIdx.x;
    int start = phase & 1;
    int j     = start + 2 * idx;
    if (j < n - 1) {
        if (d_data[j] > d_data[j + 1]) {
            int tmp       = d_data[j];
            d_data[j]     = d_data[j + 1];
            d_data[j + 1] = tmp;
        }
    }
}
\end{lstlisting}

The host loop drives the phase iteration.

\begin{lstlisting}[caption={Host-side phase loop}]
for (int phase = 0; phase < n; phase++) {
    oddeven_phase_multiblock<<<blocks, threads_per_block>>>(d_data, n, phase);
}
cudaDeviceSynchronize();
\end{lstlisting}

\subsubsection{Synchronization}

Global synchronization between phases is achieved implicitly by the CUDA kernel launch boundary. The GPU executes all threads of kernel launch $k$ before kernel launch $k+1$ begins. No synchronization is required inside the kernel, so the computation can scale freely based on the number of blocks needed.

\subsubsection{Data Decomposition}

The number of thread blocks is computed to cover all active pairs

\begin{align}
    \texttt{threads\_per\_block} &= 1024, \\
    \texttt{pairs} &= \lceil n/2 \rceil, \\
    \texttt{blocks} &= \lceil \texttt{pairs} / \texttt{threads\_per\_block} \rceil.
\end{align}

Each thread is assigned a unique global index \texttt{idx} and computes its pair position as $j = \text{start} + 2\cdot\text{idx}$. A bounds check \texttt{j < n-1} discards surplus threads in the last block, ensuring correctness when $n$ is not evenly divisible by twice the thread count.

\subsection{Correctness Validation}

The parallel implementation is verified by using the is\_sorted\_check shown below, which is done once the parallel implementation is done with the sorting. There is then a print that states if the data is sorted or not.

\begin{lstlisting}[caption={is\_sorted\_check}]

\end{lstlisting}

\section{Pearson Correlation Coefficient}

\subsection{Parallelization Strategy}

All three steps are parallelized using separate CUDA kernels. Step 1 and 2 are decompose the work by row, assigning one thread per row. Step 3 uses a two-dimensional decomposition, blocks iterate over \texttt{sample1} rows and threads within each block iterate over \texttt{sample2} values, computing one dot-product per thread.

\subsubsection{Kernel 1: Row Means (\texttt{kernel\_means})}

Each thread computes the mean of one row by iterating over all columns:

\begin{lstlisting}[caption={Row-mean kernel}]
__global__ void kernel_means(const double *matrix, double *mean,
                              int rows, int cols)
{
    int row = blockIdx.x * blockDim.x + threadIdx.x;
    if (row >= rows) return;
    double sum = 0.0;
    for (int j = 0; j < cols; j++)
        sum += matrix[row * cols + j];
    mean[row] = sum / cols;
}
\end{lstlisting}

Launched with $\lceil m / 256 \rceil$ blocks of 256 threads, this kernel is fully parallel across rows with no inter-thread communication.

\subsubsection{Kernel 2: Mean-Subtracted Matrix and Standard Deviation (\texttt{kernel\_mm\_std})}

Each thread subtracts the row mean from every element in its assigned row, accumulates the squared deviations, and writes the row standard deviation:

\begin{lstlisting}[caption={Mean-subtraction and standard deviation kernel}]
__global__ void kernel_mm_std(const double *matrix, const double *mean,
                               double *mm, double *std_dev,
                               int rows, int cols)
{
    int row = blockIdx.x * blockDim.x + threadIdx.x;
    if (row >= rows) return;
    double sum = 0.0;
    for (int j = 0; j < cols; j++) {
        double diff        = matrix[row * cols + j] - mean[row];
        mm[row * cols + j] = diff;
        sum               += diff * diff;
    }
    std_dev[row] = sqrt(sum);
}
\end{lstlisting}

Like \texttt{kernel\_means}, this kernel is embarrassingly parallel across rows and requires no synchronization.

\subsubsection{Kernel 3: Pair-wise Pearson Correlation (\texttt{kernel\_pearson})}

The correlation kernel uses a two-level loop to cover the upper triangle of the $m \times m$ pair matrix. Blocks iterate over \texttt{sample1} rows (striding by \texttt{gridDim.x} to handle cases where the number of rows exceeds the block count), and threads within each block stride over the corresponding \texttt{sample2} values:

\begin{lstlisting}[caption={Pearson correlation kernel}]
__global__ void kernel_pearson(const double *mm, const double *std_dev,
                                double *output, int rows, int cols)
{
    for (int sample1 = blockIdx.x; sample1 < rows - 1; sample1 += gridDim.x) {
        int tri_offset = (sample1 + 1) * (sample1 + 2) / 2;
        int num_pairs  = rows - sample1 - 1;
        for (int idx = threadIdx.x; idx < num_pairs; idx += blockDim.x) {
            int sample2 = sample1 + 1 + idx;
            double sum  = 0.0;
            for (int k = 0; k < cols; k++)
                sum += mm[sample1 * cols + k] * mm[sample2 * cols + k];
            output[sample1 * rows + sample2 - tri_offset] =
                sum / (std_dev[sample1] * std_dev[sample2]);
        }
    }
}
\end{lstlisting}

The triangular offset \texttt{tri\_offset} $= \frac{(\texttt{sample1}+1)(\texttt{sample1}+2)}{2}$ maps the 2D pair index $(\texttt{sample1}, \texttt{sample2})$ into the compact 1D output array of size $\frac{m(m-1)}{2}$, avoiding storage of the redundant lower
triangle.

The kernel is launched with $\min(m-1,\,256)$ blocks and 256 threads per block. For
matrices with more than 256 rows, the outer \texttt{gridDim.x} stride ensures that every
\texttt{sample1} row is covered.

\subsection{Synchronization}

No explicit synchronization is required within any kernel. Step 1, 2, and 3 are separated by implicit kernel launch barriers. So \texttt{kernel\_mm\_std} cannot begin before the \texttt{kernel\_means} has completed. CUDA guarantees sequential execution of kernel launch on the same stream, so no explicit \texttt{cudaDeviceSynchronize()} is needed between the steps.

\subsection{Data Decomposition}

Step 1 and 2 have the rows divided into contiguous blocks of 256. Each thread owns exactly one row.

In step 3 each block owns \texttt{sample1} rows via a strided grid loop. Each thread within each block owns \texttt{sample2} values via a strided thread loop. Each output cell is written by exactly one thread, so no atomic or locking operations are required.

\subsection{Correctness Validation}

The provided \texttt{verify} program compares the floating-point output files of the sequential and parallel pcc. The parallel version passes all tests for the given seed.

\section{Discussion}
The benchmarking results for the Odd-Even sort reveal a significant performance gap between the single-block and multi-block implementations.

The single-block version is constrained by the maximum number of threads per block. While it avoids the overhead of repeated kernel launches, it cannot utilize the full parallel capacity of the GPU.. As shown in the results, the speedup for $2^{19}$ elements plateaus around 5x, as most of the GPU remains idle.

The multi-block version achieves a massive speedup (up to 39x). By launching a kernel for each phase, we distribute the workload across all available blocks/threads. Although kernel launch overhead exists, it is negligible compared to the massive parallelism gained when sorting large datasets.

The PCC implementation demonstrates the point where GPU acceleration and overhead is negligible.

For small matrices ($64 \times 64$), the sequential version is actually faster (0.08x speedup). This is due to the overhead of memory allocation, data transfer, and kernel launch latency. We reach peak efficiency and the highest speedup at $1024 \times 1024$. Beyond this point ($4096 \times 4096$), the speedup begins to decrease. This is likely due to memory bandwidth saturation and the increased complexity of the $O(m^2 \cdot n)$ operation.

In both algorithms we have memory considerations, performance is heavily influenced by memory access patterns. In PCC, the use of double precision provides accuracy but consumes more bandwidth than float. The current implementation relies on global memory and further optimization could involve using Shared Memory to cache rows of the mean-subtracted matrix during the correlation step to reduce redundant global memory reads.
Another improvement to speedup could be to run stage 1 and 2 of the PCC at the same time with the same kernel without loading the data twice.

\section{Conclusion}
This assignment successfully demonstrated the application of CUDA to parallelize two types of workloads.

The Odd-Even Sort highlighted the use of multi-block decomposition for scalability. While the algorithm itself has a high time complexity of $O(n^2)$, the GPUs ability to perform thousands of comparisons simultaneously reduced execution time from over 130 seconds to under 4 seconds for large datasets.

The Pearson Correlation Coefficient demonstrated the power of the GPU for compute intensive statistical tasks. By decomposing the problem into three distinct stages, means, standard deviations, and pair-wise products we achieved a speedup of over 45x on medium sized matrices. Ultimately, the results confirm that while GPU acceleration introduces overhead, it is a useful tool for data-parallel problems, provided the problem size is large enough to saturate the hardware's massive core count.
